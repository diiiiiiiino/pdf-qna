## 실행 절차

이 스킬은 **2단계**로 실행한다.

### 1단계: 파일 읽기 + 챕터 식별 (순차)

1. 사용자의 프롬프트에 대상 파일 경로가 포함되어 있으면 그대로 사용한다. 없으면 사용자에게 묻는다.
2. 대상 파일의 총 라인 수를 확인한다. (`wc -l`)
3. 파일에서 챕터 경계를 식별한다. **목차(Table of Contents)를 우선 활용**한다.
    - 파일 앞부분(1~300줄 정도)을 Read로 읽어 목차 영역을 찾는다. (`목차`, `CONTENTS`, `책의 구성`, `Table of Contents` 등의 키워드)
    - 목차에서 챕터 이름과 번호 목록을 추출한다. (예: 2장 느려진 서비스, 3장 성능을 좌우하는 DB 설계 등)
    - 추출한 각 챕터 번호로 `^Chapter 0?{번호}$` 또는 `^{번호}장 ` 패턴을 Grep하여 본문에서의 실제 시작 라인을 찾는다.
      - 같은 패턴이 여러 번 나타나면(페이지 헤더 반복) **첫 번째 출현**을 챕터 시작으로 사용한다.
    - 목차가 없는 경우에만 `^Chapter`, `^\d+장`, `^# ` 등의 패턴으로 직접 챕터를 식별한다.
4. 각 챕터의 **이름, 시작 offset, 끝 offset(limit)**을 정리한다. 마지막 챕터의 끝은 파일 끝(총 라인 수)으로 설정한다.
5. 사용자의 프롬프트에서 Notion 업로드 지시사항(Notion 페이지 URL 등)을 추출한다.

### 2단계: 서브에이전트 병렬 실행

- 각 챕터마다 **Agent 도구**를 사용하여 서브에이전트를 실행한다.
- **모든 챕터의 Agent 호출을 하나의 메시지에서 동시에 실행**하여 병렬 처리한다.
- 각 서브에이전트는 **QnA 내용을 생성하여 바로 Notion에 업로드**한다. 로컬 파일은 생성하지 않는다.
- 각 서브에이전트에게 전달할 프롬프트에는 다음 정보를 포함한다:
    - 입력 파일 절대 경로
    - 해당 챕터의 offset과 limit (Read 도구로 읽을 범위)
    - 챕터 이름
    - QnA 생성 규칙과 템플릿
    - Notion 업로드 지시사항

### 주의사항
- 서브에이전트는 `subagent_type`을 `general-purpose`로 사용한다.
- 챕터 내용이 길면 offset/limit를 나누어 여러 번 Read할 수 있도록 프롬프트에 안내한다.
- 로컬 파일은 생성하지 않는다. Notion에만 업로드한다.
- 모든 Agent 호출이 완료되면 업로드된 챕터 목록을 사용자에게 보여준다.
- 사용자 프롬프트에 Notion 업로드 지시사항이 없으면 작업 2 대신 로컬 파일로 생성한다.
